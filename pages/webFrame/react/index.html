React

用于构建用户界面的 JavaScript 库，组件化、声明式、一次学习随处编写。以下是 React 面试高频知识点。

高频面试题

01React 的虚拟 DOM 是什么？Diff 算法怎么工作？▼

虚拟 DOM：用 JS 对象描述真实 DOM，状态变化时生成新 VNode，与旧 VNode 对比（Diff），最后只更新差异部分，减少直接 DOM 操作。

Diff 算法策略（O(n) 复杂度）：

  • 同层比较：只比较同一层级的节点
  • 类型不同直接替换：不同类型的元素（如 div → span）直接销毁重建
  • key 标识：列表用 key 复用节点，减少创建/销毁
// 虚拟 DOM 结构
{
  type: 'div',
  props: { className: 'box' },
  children: [
    { type: 'h1', props: {}, children: 'Hello' }
  ]
}
面试要点：虚拟 DOM 的核心价值不在于比直接操作 DOM 快，而在于声明式开发和跨平台能力。
02React 常用的 Hooks 有哪些？▼
  • useState：状态管理
  • useEffect：副作用（订阅、请求、DOM 操作）
  • useRef：获取 DOM 引用 / 存储可变值
  • useMemo：缓存计算结果
  • useCallback：缓存函数引用
  • useContext：跨组件状态共享
  • useReducer：复杂状态管理（类 Redux）
  • useLayoutEffect：同步执行副作用（DOM 测量）
function App() {
  const [count, setCount] = useState(0);
  const inputRef = useRef(null);

  useEffect(() => {
    document.title = `点击了 ${count} 次`;
    return () => console.log('cleanup'); // 卸载时执行
  }, [count]); // 依赖数组

  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);

  return <input ref={inputRef} />;
}
03setState 是同步还是异步的？▼

React 18 之前：在合成事件和生命周期中是异步的（批量更新），在 setTimeout/Promise 中是同步的。

React 18 起：所有 setState 都是自动批量更新（Automatic Batching），无论在哪里调用。

function handleClick() {
  setCount(1);   // 不会立即更新
  setCount(2);   // 批量合并
  console.log(count); // 还是旧值
}

// 多次 setState 会被合并为一次渲染
本质：setState 本身不是异步的，是 React 故意延迟更新以批量优化渲染性能。可以用函数式更新获取最新值：setCount(prev => prev + 1)
04useEffect 和 useLayoutEffect 的区别？▼
  • useEffect：在浏览器绘制后异步执行，不阻塞页面渲染。适合大多数场景
  • useLayoutEffect：在DOM 更新后、绘制前同步执行，会阻塞渲染。适合需要读取 DOM 布局信息并立即修改的场景
// useEffect：请求、订阅、日志
useEffect(() => {
  fetch('/api').then(...);
}, []);

// useLayoutEffect：DOM 测量后立即修改
useLayoutEffect(() => {
  const height = ref.current.offsetHeight;
  if (height > 500) {
    ref.current.style.height = '500px';
  }
}, []); // 在用户看到闪烁前完成修改
优先使用 useEffect，只有遇到闪烁问题时才用 useLayoutEffect。
05React 组件间通信有哪些方式？▼
  • 父→子：props
  • 子→父：父组件传回调函数，子组件调用
  • 兄弟组件：状态提升到共同父组件
  • 跨层级：Context（createContext + useContext）
  • 全局状态：Redux / Zustand / Jotai
  • 事件通信：EventEmitter / 自定义事件
  • ref 转发：useImperativeHandle + forwardRef
// Context 示例
const ThemeContext = createContext('light');

// 父组件提供
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>

// 子组件消费
function Child() {
  const theme = useContext(ThemeContext);
  return <div className={theme} />;
}
06受控组件和非受控组件的区别？▼

受控组件：表单值由 React state 控制，每次输入触发 state 更新。

// 受控组件
const [value, setValue] = useState('');
<input value={value} onChange={e => setValue(e.target.value)} />

非受控组件：表单值由 DOM 自己管理，用 ref 获取。

// 非受控组件
const inputRef = useRef();
<input ref={inputRef} defaultValue="默认值" />
// 需要时: inputRef.current.value
推荐使用受控组件，数据流清晰。非受控组件适合简单场景或集成非 React 代码时。
07说说 Redux 的数据流▼

Redux 是单向数据流的状态管理方案：

数据流：View → dispatch(action) → Reducer → New State → View 更新

// 1. 定义 Action Type
const INCREMENT = 'INCREMENT';

// 2. 创建 Action
const increment = () => ({ type: INCREMENT });

// 3. Reducer 处理
function counter(state = 0, action) {
  switch (action.type) {
    case INCREMENT: return state + 1;
    default: return state;
  }
}

// 4. 创建 Store
const store = createStore(counter);

// 5. 组件中使用
import { useSelector, useDispatch } from 'react-redux';
function Counter() {
  const count = useSelector(s => s);
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(increment())}>{count}</button>;
}
三大原则：单一数据源、State 只读、Reducer 纯函数。现代项目也常用 Redux Toolkit（RTK）简化写法。
08React 性能优化有哪些手段？▼
  • React.memo：缓存组件，props 不变时不重新渲染
  • useMemo：缓存昂贵计算结果
  • useCallback：缓存函数引用，配合 memo 使用
  • 懒加载：React.lazy + Suspense 按需加载组件
  • 虚拟列表：大量数据用 react-window / react-virtualized
  • key 优化：列表使用稳定唯一的 key
  • 合理拆分组件：细粒度组件 + memo 减少重渲染范围
  • 状态下放：把状态放到需要它的最小组件中
// React.memo + useCallback
const ExpensiveItem = React.memo(({ data, onClick }) => {
  return <div onClick={onClick}>{data.name}</div>;
});

function Parent() {
  const [count, setCount] = useState(0);
  const handleClick = useCallback(() => { ... }, []);
  return <ExpensiveItem data={data} onClick={handleClick} />;
}
09说说 React Fiber 架构▼

React 16 引入的可中断、可恢复的协调（Reconciliation）架构。

核心思想：将大的渲染任务拆分为多个小任务（Fiber 节点），可以在帧间暂停，让浏览器处理高优先级任务（用户输入、动画），避免长时间阻塞主线程。

  • 每个组件对应一个 Fiber 节点，形成链表树结构
  • 两个阶段：Render 阶段（可中断，计算变更）→ Commit 阶段（不可中断，执行 DOM 操作）
  • 优先级调度：高优先级任务可以插队
// Fiber 节点简化结构
{
  type: 'div',           // 组件类型
  key: null,
  child: Fiber | null,   // 第一个子节点
  sibling: Fiber | null, // 兄弟节点
  return: Fiber | null,  // 父节点
  pendingProps: {},      // 待处理 props
  flags: Placement | Update | Deletion
}
React 18 的并发特性（Concurrent Features）如 useTransition、useDeferredValue 就是基于 Fiber 架构实现的。
10React 事件机制和原生事件有什么区别？▼
  • 合成事件：React 将事件委托到根节点（React 17 前是 document，17+ 是 root），统一管理
  • 事件冒泡：React 合成事件的冒泡不是原生 DOM 冒泡，无法用 e.stopPropagation() 阻止原生事件
  • this 绑定：类组件中事件处理函数需要手动绑定 this（箭头函数或 bind）
  • 性能优化：事件委托减少了事件监听器数量
// React 合成事件
<button onClick={handleClick}>Click</button>

// React 17+ 委托在 root 上
// 原生事件先执行 → React 合成事件后执行
// (React 17 前是 document 委托，原生先到 document)

// 阻止默认行为
function handleSubmit(e) {
  e.preventDefault();
}
React 17 改变事件委托位置是为了解决多个 React 应用嵌套时的事件冲突问题。
